Q1: Open both pcaps with tcpdump -r <file> -nn -A and compare the output. List three specific things that are visible in the plaintext capture but hidden in the TLS capture. What does this mean for an attacker who can intercept network traffic between the Pi and the broker? 
ANS:
In comparing the two packet captures with tcpdump -r <file> -nn -A, the plain text capture allows you to see the topic names used for the MQTT connection, the actual payload message content sent, which in this case were the sensors' readings and test string values, and all the MQTT protocol control message commands such as CONNECT, PUBLISH, etc., in clear ASCII text format. The TLS capture, on the other hand, hides all of the above information as it is all encrypted as application data and only shows the handshake and certificates during connection establishment phase. This makes it impossible for the attacker who is intercepting the traffic to read any of the transmitted data. It also makes it possible for the attacker to launch eavesdropping attacks, man-in-the-middle attacks, and replay attacks because the connection is unencrypted.

Q2: Your mosquitto.conf sets require_certificate true. What does this mean for a client that tries to connect without a certificate? Test it: run mosquitto_sub -h <VM_IP> -p 8883 -t '#' from a terminal that does NOT provide --cafile, --cert, and --key flags. What error message do you get? Why is this a security improvement over allow_anonymous true?
ANS: 
By setting require_certificate true, you make sure that your broker will ask for an SSL certificate from each of your clients. In case you connect to the broker without any of the certificate flags set, you will be met with either a “Connection error: Connection refused” or “handshake failure” error since the connection will be rejected right away. This is far safer than allowing anonymous connections since mutual authentication is used, meaning that an attacker can never establish a connection using any other device besides yours.
Q3: Suricata is monitoring port 1883 (plaintext). It cannot read the encrypted payload on port 8883 without the TLS session keys. Given this limitation, what can Suricata still usefully observe about TLS traffic — and how would you use that to detect a threat on port 8883? This is relevant to the rules you will write in Project 2. 
ANS:
Suricata will not be able to see inside the packet’s payload but can analyze TLS handshakes such as SNI, certificate issuer information, and JA3 fingerprint. You can use this method to detect threats via rules that can trigger when a self-signed certificate is being used, a blacklisted IP address is involved, or there are abnormal volumes of traffic. The attacker's "fingerprint," along with its connection characteristics, can be detected despite the inability to analyze its payload.
