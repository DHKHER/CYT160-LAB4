Q2: The flood simulation uses paho-mqtt on port 1883 (plaintext) rather than port 8883 (TLS). Looking at your Lab 5 analysis Q3 answer, explain why Suricata can inspect the payload on port 1883 but not on port 8883. What are the trade-offs between running attacks on the plaintext versus TLS port for testing purposes?
ANS:
Because MQTT communication on port 1883 is sent in plaintext, the full packet content—including headers and message payload—is visible to the IDS, allowing Suricata to examine the payload. This enables Suricata to match patterns, apply content-based rules, and produce alerts based on the actual contents of the packets. On the other hand, MQTT traffic between the client and broker is encrypted via TLS encryption on port 8883. As a result, Suricata can only see metadata such as IP addresses, ports, and packet sizes, but not the actual payload content, so it cannot perform deep packet inspection or match payload-based signatures unless SSL/TLS decryption is configured (which is usually not done in basic lab setups).

The trade-off for testing is that plaintext (1883) allows you to see how rules are triggered based on packet content, which makes it easier to watch and validate IDS behavior. However, since encryption is usually used in real-world systems, this is less practical from a security perspective. However, TLS (8883) restricts visibility for tools like Suricata, making it more difficult to identify payload-based attacks, but it better represents real-world secure deployments and protects data in transit. Thus, TLS is more suited for practical security scenarios but necessitates more complex configurations (like decryption) for complete inspection, whereas plaintext is helpful for learning and testing detection logic.

Q3: SID 2000003 uses depth:100. The malformed script sends 'A'*400. Would the rule still fire if the repeated pattern started at byte 200 of the payload? Explain what depth: means and how you would change the rule to detect a late-starting pattern.
ANS:
No, the rule would not apply if the repeated pattern started at byte 200. The depth:100 keyword tells Suricata to only look at the first 100 bytes of the payload when searching for the specified content match. Because the dangerous pattern ('A'*400) begins at byte 200 and is completely outside the range under investigation, the rule will not detect it.

The depth option, which is helpful for performance optimization and for focusing on patterns anticipated close to the start of a packet, basically restricts how deep into the payload Suricata will look for a pattern. Threats that emerge later in the payload, however, may go unnoticed.

You might change the rule to either increase the depth (depth:400 or greater) so that more of the payload is examined or remove the depth choice completely to enable Suricata to scan the entire payload in order to find a pattern that begins later. Another way to target patterns that start later in the payload is to combine offset and depth (e.g., offset:200; depth:400;).
